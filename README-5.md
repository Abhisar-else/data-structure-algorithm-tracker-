# Algorithm Flight Recorder
### WebAssembly + C++ + SQLite + Vanilla JS

A browser-based algorithm visualizer where the **real computation happens in compiled C++**, recorded step-by-step into a visual timeline, with performance metrics persisted to an embedded SQLite database.

---

## Architecture

```
[ Browser UI ]  ◄── (animates step timeline) ──┐
      │                                         │
 (user input)                                   │
      ▼                                         │
[ JavaScript Bridge ]  ──(array data)──►  [ C++ / WASM Engine ]
                                                │         │
                                         (records steps)  (writes metrics)
                                                │         │
                                           [ Steps[] ]  [ SQLite DB ]
```

### Three Layers

| Layer | File | Role |
|-------|------|------|
| **Frontend** | `public/index.html` | UI, bar chart renderer, playback controls |
| **Bridge** | `public/engine.sim.js` / `engine.wasm-bridge.js` | Hands data to engine, receives step reel |
| **C++ Engine** | `src/engine.cpp` | Runs algorithms, records every micro-decision, logs to SQLite |

---

## Algorithms Implemented

| Algorithm | Complexity | Notes |
|-----------|-----------|-------|
| Bubble Sort | O(n²) | Swap-optimized with early exit |
| Selection Sort | O(n²) | Tracks minimum pointer each pass |
| Insertion Sort | O(n²) | In-place, stable |
| Merge Sort | O(n log n) | Recursive divide-and-conquer, full merge trace |
| Quick Sort | O(n log n) | Lomuto partition, pivot highlighted |
| Binary Search | O(log n) | Auto-sorts input, records range narrowing |

Each algorithm records:
- **Full array snapshot** at every decision point
- **Active indices** (primary = red, secondary = yellow, pivot = purple)
- **Phase label**: `compare | swap | pivot | merge | done`
- **Human-readable action string**

```bash

cd public
python3 -m http.server 8080

```

The JS simulation (`engine.sim.js`) exposes the **identical API** to the real WASM build, so the UI works immediately.

## Building the Real C++ WASM Engine

### 1. Install Emscripten

```bash
git clone https://github.com/emscripten-core/emsdk.git
cd emsdk
./emsdk install latest
./emsdk activate latest
source ./emsdk_env.sh   # add to ~/.bashrc for persistence
```

### 2. Compile

```bash
chmod +x build.sh
./build.sh
```

This outputs:
- `public/engine.js`   — Emscripten JS loader
- `public/engine.wasm` — Compiled C++ binary

### 3. Swap the bridge

In `public/index.html`, replace:
```html
<script src="engine.sim.js"></script>
```
With:
```html
<script src="engine.js"></script>
<script src="engine.wasm-bridge.js"></script>
```

The UI automatically detects the live WASM engine and updates the status indicator.

## SQLite Integration

The C++ engine maintains an in-memory SQLite database (`:memory:` in dev; swap to a file path or Emscripten's IDBFS for persistence across page loads).

**Schema:**
```sql
CREATE TABLE history (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  timestamp   TEXT DEFAULT (datetime('now')),
  algorithm   TEXT,
  input_size  INTEGER,
  total_steps INTEGER,
  comparisons INTEGER,
  swaps       INTEGER
);
```

Every time you run an algorithm, a row is written. The **History Log** tab reads this table and renders it — giving you a cross-algorithm performance comparison over time.

## Extending the Project

### Add a new algorithm

1. Implement it in `engine.cpp` using the `record()` helper
2. Add it to the `switch` in `run_algorithm()`
3. Mirror the same logic in `engine.sim.js` for the JS fallback
4. Add a button in the `algo-grid` in `index.html`

### Persistent SQLite (survive page reload)

Replace `:memory:` with Emscripten's IDBFS:
```cpp
sqlite3_open("/data/recorder.db", &g_db);
```
And mount the filesystem in JS:
```javascript
FS.mkdir('/data');
FS.mount(IDBFS, {}, '/data');
FS.syncfs(true, callback); // load
// after writes: FS.syncfs(false, callback); // save
```

### Add a tree / linked list visualizer

The step recorder is data-structure agnostic — just change the `array` field in `Step` to a different serialization format and update the renderer in `index.html`.

## What You Learn

- **WebAssembly**: How C++ compiles to `.wasm` and runs at near-native speed in the browser with no server
- **Low-level data tracing**: Every pointer move, swap, and comparison is a recorded event you can scrub through
- **Emscripten interop**: `ccall`, `cwrap`, manual heap allocation (`_malloc`/`_free`), string marshalling
- **Embedded databases**: SQLite running inside a compiled binary inside a browser tab
- **Algorithm analysis**: Comparisons vs swaps vs total steps across algorithms become concrete numbers you generated yourself
